## 1. Feature Description

### Overview
The Loan Input Form is the primary user interface component that allows consumers to enter loan parameters for calculation. This React component provides three input fields with client-side validation, clear labeling, and user-friendly error messaging. The form serves as the entry point for the entire loan simulation experience.

On **Desktop** and **Tablet** viewports, the form is rendered inside a persistent left side-panel that remains visible alongside the calculation results. On **Mobile** viewports, the form occupies the full width and stacks above the results.

### Context
As the first touchpoint in the user journey, this form must be intuitive, accessible, and provide immediate feedback on input validity. It captures the three essential parameters (loan amount, term in months, interest rate) and communicates with the backend API to retrieve calculation results. The side-panel layout on larger screens allows users to adjust inputs and see updated results without scrolling back to the form.

### Business Value
- Enables users to easily input loan parameters without confusion
- Reduces errors through client-side validation before API calls
- Provides clear guidance on expected input formats
- Creates a positive first impression for the educational tool

## 2. User Stories

### Story 2.1: Enter Loan Parameters
**As a** consumer exploring loan options  
**I want to** enter my loan amount, term, and interest rate in a simple form  
**So that** I can see how my monthly payments will be calculated

**Acceptance Criteria**:
- Form displays three clearly labeled input fields
- Each field accepts numeric input with appropriate formatting
- Form is visually clean and easy to understand
- Submit button is clearly visible and accessible

---

### Story 2.2: Receive Input Validation Feedback
**As a** consumer entering loan information  
**I want to** receive immediate feedback if my input is invalid  
**So that** I can correct errors before submitting the form

**Acceptance Criteria**:
- Validation occurs on form submission
- Error messages appear near the relevant field
- Error messages clearly explain what's wrong and how to fix it
- Valid inputs do not show error messages
- Submit button is disabled or shows appropriate state during processing

---

### Story 2.3: Understand Input Requirements
**As a** consumer unfamiliar with loan terminology  
**I want to** see helpful labels and placeholders for each field  
**So that** I understand what information to provide

**Acceptance Criteria**:
- Each field has a descriptive label
- Placeholders show example values or format hints
- Interest rate field clarifies it's an annual percentage
- Term field specifies it's in months
- Tooltips or help text available for complex fields (optional enhancement)

## 3. Functional Requirements

### 3.1 Form Fields

#### Field 1: Loan Amount
- **Label**: "Loan Amount" or "Principal Amount"
- **Input Type**: Number (decimal allowed)
- **Placeholder**: "e.g., 10000"
- **Format**: Accept numeric input, optionally with currency symbol
- **Validation**:
  - Required field
  - Must be greater than 0
  - Must be a positive number
  - Maximum reasonable value: 10,000,000 (optional limit)
- **Display**: Format with thousand separators if possible (e.g., $10,000)

#### Field 2: Loan Term
- **Label**: "Loan Term (months)"
- **Input Type**: Number (integer only)
- **Placeholder**: "e.g., 12"
- **Format**: Accept whole numbers only
- **Validation**:
  - Required field
  - Must be greater than 0
  - Must be a positive integer
  - Reasonable range: 1-600 months (50 years max)
- **Note**: Explicitly indicate "months" in label to avoid confusion

#### Field 3: Annual Interest Rate
- **Label**: "Annual Interest Rate (%)"
- **Input Type**: Number (decimal allowed)
- **Placeholder**: "e.g., 5.0"
- **Format**: Accept decimal numbers
- **Validation**:
  - Required field
  - Must be greater than or equal to 0
  - Reasonable range: 0-100 (percentage)
  - Allow up to 2 decimal places for precision
- **Display**: Show "%" symbol in label or as input suffix

---

### 3.2 Form Behavior

#### Submission
- **Submit Button**: 
  - Label: "Calculate" or "Calculate Payment"
  - Position: Below input fields, centered or right-aligned
  - State management:
    - Default state: Enabled when form is ready
    - Loading state: Show loading indicator during API call
    - Disabled state: When form has validation errors or is processing
  
#### Validation Timing
- **Client-side validation**: Triggered on form submission (before API call)
- **Optional**: Real-time validation on blur (when user leaves a field)
- **API validation errors**: Display errors returned from backend

#### Form Reset
- **Clear/Reset Button** (optional): Allow users to clear all fields and start over
- **After successful submission**: Option to modify inputs and recalculate

---

### 3.3 Error Handling

#### Validation Error Display
- **Location**: Inline errors displayed near each field with issues
- **Visual Style**: Red text or border to indicate error state
- **Error Messages**:
  - Loan Amount:
    - "Please enter a loan amount"
    - "Loan amount must be greater than 0"
    - "Please enter a valid number"
  - Loan Term:
    - "Please enter a loan term"
    - "Loan term must be at least 1 month"
    - "Please enter a whole number for months"
  - Interest Rate:
    - "Please enter an interest rate"
    - "Interest rate must be between 0 and 100"
    - "Please enter a valid percentage"

#### API Error Handling
- **Network errors**: "Unable to connect to server. Please try again."
- **Server errors**: "An error occurred while calculating. Please try again."
- **Validation errors from API**: Display specific error messages from backend

### 3.6 Responsive Layout Behavior

The LoanForm component adapts its container and layout based on three responsive breakpoints. Per the style guide, the overall app uses a two-column layout with a narrow vertical navigation sidebar on the far left (~76px, purple `#5B4FFF` background).

#### Desktop (`≥1024px`) — Left Side-Panel
- **Container**: Rendered inside a fixed-width left side-panel (~400-450px per style guide)
- **Position**: Left side of the viewport (to the right of the vertical navigation), alongside the main content area
- **Behavior**: Panel remains visible and accessible while the user scrolls through results
- **Form Layout**: Fields stack vertically within the side-panel, inside a white card with shadow
- **Submit Button**: Full width within the panel
- **Card Styling**: White background, shadow (`0 2px 8px rgba(0,0,0,0.08)`), rounded corners (12-16px), padding (32-40px)

#### Tablet (`≥768px and <1024px`) — Narrower Left Side-Panel
- **Container**: Rendered inside a narrower left side-panel (~300-350px)
- **Position**: Left side of the viewport (to the right of the vertical navigation), alongside the main content area
- **Behavior**: Same persistent panel concept as Desktop but with reduced width
- **Form Layout**: Fields stack vertically within the side-panel
- **Submit Button**: Full width within the panel
- **Adjustments**: Labels may wrap; input fields use full panel width
- **Card Styling**: Same white card treatment as Desktop

#### Mobile (`<768px`) — Full-Width Stacked
- **Container**: Full viewport width, stacked above results
- **Position**: Top of the page, above PaymentDisplay and InstallmentTable
- **Behavior**: Standard vertical flow; user scrolls down to see results
- **Form Layout**: Fields stack vertically, full width, inside a white card
- **Submit Button**: Full width
- **Vertical Navigation**: Hidden or collapsed on mobile

**Implementation Notes**:
- Use CSS media queries or container queries to switch between side-panel and stacked layouts
- The side-panel layout is managed by the parent `App` component; the `LoanForm` component itself is layout-agnostic and fills its container
- On Desktop/Tablet, the side-panel should have a subtle visual separator (border or shadow) from the main content area
- App background color: Light gray (`#F5F5F7`) per style guide
- All spacing between cards: 24px gaps per style guide

### 3.7 API Integration

#### API Call
- **Endpoint**: `POST http://localhost:8000/api/calculate-loan`
- **HTTP Client**: Fetch API (native browser API)
- **Request Format**:
```json
{
  "amount": 10000,
  "term": 12,
  "rate": 5.0
}
```

#### Request Handling
1. Validate form inputs on client-side
2. If valid, send POST request to backend
3. Show loading state during request
4. On success: Pass response data to parent component for display
5. On error: Display appropriate error message

#### Loading State
- Show loading indicator (spinner or text) during API call
- Disable submit button to prevent duplicate submissions
- Optional: Show "Calculating..." message

## 6. Constraints

### Technical Constraints
- **Client-side validation only**: No server-side validation during input (only on submission)
- **No data persistence**: Form data is not saved between sessions
- **Single form instance**: No support for multiple loan comparisons
- **Synchronous submission**: Form waits for API response before allowing new submission

### Functional Constraints
- **Term unit fixed**: Only accepts months (no year option)
- **No advanced features**: No sliders, no real-time calculation preview
- **Basic error handling**: Simple error messages only
- **No field suggestions**: No autocomplete or suggested values

### Design Constraints
- **Side-panel layout**: On Desktop and Tablet, the form is contained within a left side-panel managed by the parent App component
- **Full-width on Mobile**: On Mobile (<768px), the form switches to a standard full-width stacked layout
- **Minimal styling**: Focus on functionality over aesthetics for PoC
- **Standard inputs**: Use native HTML input elements (no custom components required)
- **Layout-agnostic component**: The LoanForm component itself fills its container; responsive layout switching is handled by the parent
